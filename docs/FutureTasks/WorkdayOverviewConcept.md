# Workday Overview UI Concept

## Goal

Design a dashboard view that gives the user a fast, honest understanding of how the current day actually went:

- what the highest-focus, highest-impact blocks were
- when the user was meaningfully working
- when the user drifted, stopped, or took breaks
- when pauses happened
- why the system believes those judgments are correct

This should fit the current product shape: a local-first, explainable, self-authored focus coach rather than a surveillance dashboard.

## Why This Fits The Current Architecture

The existing architecture already supports most of the right ingredients:

- the dashboard is the correct secondary surface for day understanding
- the history/review module already expects a `day timeline`
- the logic layer is the single source of truth for classification, progress, explainability, and cooldowns
- the evening debrief packet already depends on aligned blocks, support work, ambiguous windows, pauses, and estimate-vs-actual signals

So this feature should not be a separate analytics subsystem. It should be a richer day-level dashboard slice backed by the same logic engine and persisted evidence model.

## Product Principles

The overview should follow these rules:

1. Show the day as it happened, not as a moral score.
2. Separate `aligned work`, `support work`, `drift`, `pause`, and `inactive/not observed` states clearly.
3. Distinguish `focus` from `impact`.
4. Always expose why a block was labeled a certain way.
5. Never let the UI infer business meaning from raw gaps or timestamps alone.
6. Preserve user autonomy: pauses and intentional detours must remain visible as valid choices, not failures.

## Core User Questions

This screen should answer, within a few seconds:

- When did I actually work today?
- Which stretches were my strongest focus blocks?
- Which stretches likely mattered most for progress?
- Where did the day fragment?
- When did I pause on purpose?
- Where is the evidence weak or ambiguous?

## Proposed Screen: `Workday Overview`

This should live inside the dashboard/history area as a dedicated day review screen for `today`, and later become reusable for past days.

### 1. Header Summary

Top summary row with 4-5 compact cards:

- `Focused work`: total aligned time today
- `Support work`: total valid non-primary work today
- `Drift / off-plan`: total probable non-goal time
- `Paused / away`: explicit pauses plus inferred break windows
- `Day quality summary`: one sentence generated by logic, for example:
  - `Two strong focus blocks on Task 1. Afternoon fragmented after 14:30.`

This row should be descriptive, not gamified.

### 2. Full-Day Timeline

The main visual should be a horizontal timeline spanning the full workday.

Each segment should be rendered as a continuous block with color and pattern:

- `green`: aligned work
- `blue`: support work
- `yellow`: uncertain / ambiguous
- `red`: hard drift
- `gray solid`: explicit pause
- `gray light / dashed`: inactive, idle, or insufficient evidence

The timeline should show:

- hour markers
- major transitions
- pause boundaries
- optional focus intensity overlay
- optional milestone/event pins

This is the primary answer to `when did I work and not work?`

### 3. Top Blocks Strip

Above or below the main timeline, show two ranked horizontal strips:

- `Highest Focus Blocks`
- `Highest Impact Blocks`

Each item is a compact pill/card with:

- time range
- task name
- duration
- score label
- one-line evidence summary

Example:

- `09:10-10:35  Task 1  85m  High focus  Stable aligned evidence, low switching`
- `11:20-12:05  Task 2  45m  High impact  Commit + milestone confirmation`

This is important because the user asked both about `high focus` and `most impact`, and those are not always the same block.

### 4. Task Lanes

Below the main timeline, render one lane per planned task for the current day.

Each lane should show:

- all blocks matched to that task
- support work attached to that task
- progress markers
- risk marker if the task ended behind intended pace

This helps answer: `Where did my time go relative to the morning contract?`

### 5. Pause And Break Lane

Render a dedicated narrow lane for:

- explicit user pauses
- accepted breaks
- long inactive windows
- intentional detours if user confirmed them as valid

This should make breaks legible instead of burying them inside general gray gaps.

### 6. Explainability Drawer

Clicking any segment should open a right-side detail drawer with:

- state label
- matched task
- confidence ratio
- top evidence
- whether it counted as support work
- whether the user corrected it later
- artifact or milestone evidence if present

This preserves the architecture rule that the system must explain `why`.

## Focus vs Impact

The interface should explicitly model these as different dimensions.

### High Focus

A `high focus` block is a block with:

- sustained aligned classification
- low switching / interruption
- low ambiguity
- strong confidence
- meaningful duration

This is about concentration quality.

### High Impact

A `high impact` block is a block that materially moved a planned task forward.

Signals can include:

- milestone completion
- artifact creation or meaningful edits
- commit or document progress
- movement on a high-risk or high-priority task
- confirmed user correction that reclassified the block as valuable support work

This is about outcome contribution, not just time.

Important: `impact` scoring should be produced by the logic layer, not inferred in SwiftUI.

## Needed Data Shape Extension

The current contract has `recent_episodes`, task progress cards, and corrections, which is not enough for a strong full-day visualization.

Add a dedicated day-level view model to the logic/UI contract, for example:

```ts
type WorkdayOverviewViewModel = {
  local_date: LocalDate;
  summary_text: string;
  totals: {
    aligned_seconds: DurationSeconds;
    support_seconds: DurationSeconds;
    drift_seconds: DurationSeconds;
    paused_seconds: DurationSeconds;
    inactive_seconds: DurationSeconds;
  };
  timeline_segments: DaySegment[];
  top_focus_blocks: RankedDayBlock[];
  top_impact_blocks: RankedDayBlock[];
  task_lanes: TaskDayLane[];
};

type DaySegment = {
  segment_id: OpaqueId;
  started_at: IsoUtc;
  ended_at: IsoUtc;
  visual_state: "aligned" | "support" | "uncertain" | "hard_drift" | "paused" | "inactive";
  matched_task_id: OpaqueId | null;
  matched_task_title: string | null;
  confidence_ratio: Ratio | null;
  explainability: ExplainabilityItem[];
  artifact_markers: DayArtifactMarker[];
  was_user_corrected: boolean;
};

type RankedDayBlock = {
  segment_id: OpaqueId;
  started_at: IsoUtc;
  ended_at: IsoUtc;
  matched_task_title: string | null;
  score_label: string;
  summary_text: string;
};
```

### Important Boundary Rule

The UI must receive already-normalized day segments from logic.

It must not:

- infer pauses from missing records
- merge adjacent episodes by itself
- derive impact from commits or milestones on its own
- reinterpret uncertain windows as drift

That would violate the existing architecture principle that the logic layer owns business meaning.

## Classification Rules For The Overview

To keep the screen trustworthy, the day should be normalized into a small set of visually stable block types:

1. `Aligned work`
   The block clearly advances one of today’s tasks.

2. `Support work`
   The block is not direct primary execution but still counts toward a planned task.

3. `Ambiguous`
   Evidence is mixed or incomplete. The UI should show this honestly instead of pretending precision.

4. `Drift`
   The block is probably off-plan or unrelated.

5. `Paused`
   Explicit pause, break, snooze, or user-confirmed intentional detour.

6. `Inactive / not observed`
   Long idle gaps, machine sleep, no meaningful event stream, or degraded sensing.

This separation matters because `not working` is not one thing.

## Interaction Details

### Hover / Selection

Hovering a segment should show:

- time span
- task
- state
- confidence
- short evidence summary

Selecting a segment should synchronize:

- main timeline highlight
- task lane highlight
- detail drawer content

### Filters

Allow lightweight filters:

- all blocks
- focus only
- impact only
- pauses only
- ambiguous only
- per task

These should only affect rendering, not meaning.

### Day Comparison Hook

Do not build week analytics yet, but leave room for:

- compare planned vs actual day
- compare today vs yesterday
- show reusable support-work patterns learned over time

## Edge Cases

The overview must handle these explicitly:

- `no_plan`: show raw day structure but label impact as unavailable or low confidence
- `paused` mode for long periods: show that as intentional pause, not failure
- `degraded_screenpipe`: show sensing gaps clearly
- midday plan reset: visually mark the reset boundary
- unresolved ambiguity: show yellow blocks with low-confidence labeling
- corrected history: preserve original block timing but reflect final accepted classification

## Why This UI Is Better Than A Simple Activity Chart

A basic productivity timeline would show motion. This concept shows meaning.

It answers:

- Was I focused?
- Did that focus matter?
- Where did the plan break down?
- Which interruptions were legitimate?
- What evidence is solid versus uncertain?

That makes it useful both for real-time dashboard reflection and for the evening debrief flow.

## Recommended Delivery Order

1. Add a normalized `WorkdayOverviewViewModel` to the logic/UI contract.
2. Build a read-only timeline with state colors and pause/inactive distinction.
3. Add ranked `top focus` and `top impact` blocks.
4. Add task lanes.
5. Add segment detail drawer with explainability.
6. Reuse the same data in the evening debrief export.

## Outcome

If done correctly, the user should be able to open this screen and understand the day in under 10 seconds:

- where the best work happened
- where the day fragmented
- where breaks happened
- what likely counted most
- what still needs clarification

That matches the current architecture, the explainability requirement, and the product’s core principle of being a calm self-authored focus coach.
